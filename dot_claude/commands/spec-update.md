仕様を作成または更新する。

## 引数

- 機能名、コマンド名、作業名を受け取る。
- 引数がなければ「どの仕様を更新しますか?」と聞く。

## 手順

1. 既存仕様を探す。
   - `docs/features/{name}.md`
   - `docs/commands/{name}.md`
   - `docs/{name}.md`
   - `docs/spec.md`
2. ユーザーの意図、既存実装、関連ドキュメントを読んで変更案を作る。
3. 不明点が成果物の方向性を変える場合だけ確認する。
4. 適切な仕様ファイルを作成または更新する。
5. `docs/spec.md` や `docs/plan.md` と重複管理している場合は同期する。
6. 変更点、未決事項、次に進める作業を短く報告する。

## 推奨フォーマット

```markdown
# {name}

## 概要

## 仕様

## 対象ファイル

## エッジケース

## 検証

## 未決事項
```

## 注意

- 仕様は実装手順ではなく、期待する状態と判断基準を書く。
- 公開 dotfiles では secret を置かない方針と private 化する値を明示する。
